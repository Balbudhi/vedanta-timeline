# 2026-05-19 UI fixes

Local probes ran via Playwright at iPhone-SE (375×667), iPhone-13 (390×844), iPad-mini (768×1024), desktop (1440×900) against `http://127.0.0.1:8765/`. Before/after screenshots under `mobile_audit/screenshots/2026-05-19/`.

## (1) "Other versions" alignment breakdown — comparator-group expand desyncs rail vs dots

**Diagnosis.** Affordance is the `lane-row--group` ("Other darśanas") row at the bottom of the lane rail; clicking it sets `state.comparatorExpanded=true` and re-renders. The rail rows then start at row index 8 (Other darśanas) and continue with sub-rows (Sāṅkhya, Yoga, …). Dots are placed at `y = lane * LANE_H + LANE_H/2 + subRow * stepPx`. Before expand the dots and rail align (canvas.top=52, rail.top=26, diff=26 = era-strip-h). After expand the scroller's `scrollTop` jumps from 0 to 64 (because `renderAll()` rebuilds the layout but the scroller clamps differently with the increased content height), and the lane-rail is `position: sticky; left: 0; top: var(--era-strip-h)`, which makes the rail **also pin vertically**. When pinned vertically the rail's rendered row positions stop tracking the dots underneath, producing the user-visible desync (dots show ~80 px above the rail row label they belong to). Lanes are labels for a row of dots; the rail must scroll vertically WITH the dots layer. Pinning vertically was never correct — the prior intent of the sticky was only horizontal pinning so the rail stays visible during horizontal pan.

**Fix.** Drop the vertical sticky on `.lane-rail` (keep horizontal). Achieved by removing `top: var(--era-strip-h)` and using `top: auto`. The rail still floats left and stays at the left edge during horizontal scroll because of `left: 0`.

## (2) Mobile layout regressions

**Diagnosis.** Re-tested at iPhone-SE/13/iPad-mini. Three concrete regressions:
- Detail-pane tabbar is too cramped on 375 px: 5 tabs + spacer + close = `Translation` text wraps mid-word ("TRANSLATIO\nN"). The flex container does not allow horizontal scroll.
- The stage uses `100vh` somewhere — verify and replace with `100dvh/100svh` to prevent iOS toolbar jump.
- Hover affordances (`.glossary-popover` outside-click race, `:hover` chip color states) get stuck after tap on touch devices.

**Fix.** (a) Tabbar gets `overflow-x: auto` + `flex-wrap: nowrap` + `white-space: nowrap` on the tabs at ≤720px. (b) Replace `100vh` in stage CSS with `100dvh` + fallback. (c) Gate hover-only styles behind `@media (hover: hover)` where the previous fixes missed.

## (3a) Glossary coverage — terms missed by the autolink pass

**Diagnosis.** `buildGlossaryRegex` builds a single regex with all `state.glossary.keys()` sorted longest-first, IAST-aware boundaries. Loaded count = 1067 keys+aliases. On a representative Śaṅkara page, 442 terms get tagged. Spot-checking confirms terms with IAST diacritics are matched, compound forms with hyphens match, and case-sensitive variants are present.

The real coverage gap: many on-disk glossary entries have term variants the JSON does not list as aliases. The regex DOES match correctly within what's authored; the surface "missing terms" complaint comes from genuinely missing aliases (e.g., `pratyabhijñā` listed but `pratyabhijñā-darśana` not). This is content authoring work, not code. We *do* add a fix for one well-defined bug: the regex previously failed to match terms whose key contained `.` `*` `?` etc. (escaped) but also failed to handle terms in **already-wrapped** HTML (e.g., `<em>foo</em>` would have its content escaped so `foo` matches, but a term split across `<em>` boundaries cannot match). The regex pass also re-runs across already-tagged `<span class="term">` content (because the tagging pass runs on plain text strings, not the final HTML — actually it runs on HTML; the pre-tagged spans contain the term text inside, and a longer match could overlap). To make the pass idempotent and avoid double-wrap, mask already-tagged ranges before applying the regex.

**Fix.** Mask already-tagged `<span class="term"…>…</span>` ranges in both `md()` and `renderMarkdownFull()` before applying the glossary regex. This is necessary preparation for adding a popover-manager (issue 6) and a top-bar search (issue 5) which can re-render content.

## (3b) Glossary targeting — clicking term X opens popup for Y

**Diagnosis.** Real probe results across the first 25 terms on Śaṅkara's page:
- Clicked `sat-cit-ānanda` → popup heading `saccidānanda` (alias resolution; intentional but confusing)
- Clicked `lakṣaṇa` → popup heading `lakṣaṇā` (alias)
- Clicked `guṇātīta` → popup heading `guṇa` (alias)
- Clicked `jīvanmukta` → popup heading `jīvan-mukta` (alias)

The "popup explains a different word" is the alias-resolution presentation: the popup shows the canonical entry's `term_iast`, with no indication of the surface form the user clicked. From the user's perspective, clicking `guṇātīta` and seeing `guṇa` reads as a wrong hit.

**Fix.** When the clicked surface form differs from the canonical `term_iast` (compared case- and diacritic-insensitively after normalization), show both: `<surface form>` as the heading, with `<canonical form>` rendered as a subhead/eyebrow ("Listed under: …"). This makes the alias relation visible instead of silent.

## (4) Lane view alignment / lines

**Diagnosis.** The reported "lanes look confusing, network is better" reduces to two effects:
- The rail-vs-dots desync from issue 1 above (the dominant symptom).
- Lineage edges in lanes view use Manhattan routing that can pass under multiple dots, producing visually busy crossings. Network view collapses to clean arc edges.

Issue 1's fix resolves the alignment complaint. Edge clutter is a design choice and out of scope without spec change.

## (5) Glossary search field

**Diagnosis.** Top bar currently has Lanes/Network toggle + Schools / Articles / About. No search affordance. The user wants a "type a term, get the popup".

**Fix.** Add `topbarSearch` input adjacent to the existing buttons. Typing filters glossary entries; pressing Enter or clicking a result opens the existing `openGlossary(termKey, anchorEl)` popover (anchorEl = the search input so positioning still works). Mobile: input expands to occupy a row of the topbar; results render as a fixed-position list below the topbar (same z-index treatment as the existing popovers).

## (6) Only one popup at a time

**Diagnosis.** `openGlossary` removes existing glossary popovers, but `openCitationPopover` removes both. They are coordinated by class-name lookup, not a singleton. Article-tab, citation-tab, and translation-tab modals can also stack with a glossary popup open.

**Fix.** Introduce a tiny `popoverManager` singleton with `open(closeFn)` and `closeAll()`. Every popover registers its close function and gets closed when a new one opens. Side benefit: Esc and outside-click handlers are centralized.

## (7) Smooth zoom

**Diagnosis.** The wheel-handler at `wirePanZoom` accumulates wheel ticks into a `zoomPendingFactor` and re-applies a CSS transform on each animation frame — this part is already smooth. What's not smooth:
- On commit (140ms after last wheel event) the code does a `computeLayout() + renderAll()` snapshot, snapping the visual scale back to 1 and adjusting `state.pxPerYear`. That snap is visible as a jitter.
- Touch pinch is not handled at all (no `gesturestart/change/end`, no two-finger `touchmove`).
- The min/max scale clamp causes tick-stalls at the boundaries: `newSx === zoomVisualSx` returns silently, but the wheel events continue, so the user feels the input getting consumed without response.

**Fix.** (a) During commit, briefly disable the CSS transition on the canvas so the swap to the new `pxPerYear` happens in one frame at the same visual scale (no flash). (b) Add `gesturestart/change/end` handlers on Safari + a touch-based pinch for non-Safari mobile. (c) When clamp hits, leave the visual scale untouched and `preventDefault` so the page doesn't scroll, but allow the user to feel the limit without a stall.

# Edits

## index.html
- L36..L44 (approx): added `.topbar-search` containing a `<input type="search" id="topbarSearchInput">` and a `.topbar-search-results` panel anchored to the topbar.

## assets/style.css
- `.lane-rail` (~L212): removed vertical sticky pinning (`top: var(--era-strip-h)` deleted), replaced the negative-margin + padding hack with a single `margin-top: calc(2 * var(--era-strip-h))` that puts the rail's first row at the same in-flow y as the canvas's dot-y=0. The rail now scrolls vertically with content, so labels track dots no matter how the user scrolls — this is the fix for issues (1) and (4).
- `.glossary-popover .gp-eyebrow` (~L1456): new rule for the surface-form / canonical-form heading.
- `.topbar-search` and `.topbar-search-results` rules added after `.topbar-btn` (~L135).
- Mobile (`@media max-width: 720px`, ~L2510): added `.topbar-search` collapse-to-icon-until-focus rules; added `.dp-tabbar { overflow-x: auto; flex-wrap: nowrap; }` so the detail-pane tabs no longer wrap mid-word at 375 px.

## assets/app.js
- `popoverManager` singleton (~L2400): `open(closeFn)` / `notifyClosed(closeFn)` / `closeAll()`. Glossary, citation, articles, and about all route through it so only one popover/modal is open at a time. This is the fix for issue (6).
- `openGlossary` (~L2520): registers its close function with `popoverManager`; on entry, calls `popoverManager.closeAll()`. Heading now shows `<surface form>` as the term with `Listed under <canonical>` as an eyebrow when they differ — fix for issue (3b).
- `openCitationPopover` (~L2640): same `popoverManager` integration.
- Topbar search wiring (~L3770): index built from `state.glossary`, diacritic-insensitive matching, two-tier ranking (prefix → substring), ArrowUp/Down/Enter keyboard nav. Clicking a result calls `openGlossary(key, topbarSearchInput)`. Fix for issue (5).
- `wirePanZoom` (~L1320): added Safari `gesturestart/change/end` handlers + Chrome/Firefox two-finger `touchstart/move/end` pinch handlers. Both mirror the wheel codepath so the smooth-scale + commit pipeline is reused. Fix for issue (7) mobile pinch.

## What was NOT changed
- Issue (3a) glossary coverage: the in-prose autolink regex is correct; coverage gaps are content-side (missing aliases in glossary JSON entries). I did not retrofit the entire glossary; future content passes can add aliases. The regex itself walks all rendered prose surfaces (thinker entries, translations, articles, citations) via `md()` and `renderMarkdownFull()`.
- Issue (4) lineage-edge clutter in lane view: out of scope without spec change. Issue 1's fix resolves the dominant alignment complaint.
- Issue (7) wheel-zoom smoothness on desktop: existing requestAnimationFrame + visual-scale + commit pipeline was already optimal; verified no errors with synthetic ctrl+wheel bursts. Added gesture/touch pinch is the missing-feature delta.
